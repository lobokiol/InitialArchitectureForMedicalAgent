# 面试问题汇总 002 - AI Agent 工程师专项

> 基于项目实战经验，按"结论 → 设计 → 关键细节 → 工程点"结构整理

---

## 一、Agent 架构设计 (5题)

### 1. Agent 系统整体架构

**结论：**
按"**感知 → 决策 → 执行 → 记忆 → 反馈**"五层设计。

**设计：**
- 感知层：用户输入 + query rewrite
- 决策层：意图识别 + 路由（RAG / Tool / Direct LLM）
- 执行层：RAG pipeline + Tool calling
- 记忆层：短期（对话历史）+ 长期（Checkpoint 持久化）
- 反馈层：流式输出 + 日志 + 评估

**关键细节：**
- 每一步都是可插拔节点（LangGraph StateGraph）
- 所有路径都有 fallback（降级策略）
- 意图识别用 7B INT4，延迟<200ms

**工程点：**
- 状态存 Redis + LangGraph Checkpoint
- 支持异步 + SSE streaming
- 条件边实现动态路由

---

### 2. 为什么选 LangGraph 而不是 AutoGPT/LlamaIndex？

**结论：**
LangGraph 提供**确定性状态机**，适合医疗导诊这种需要严格流程控制的场景。

**设计：**
- LangGraph：有向图 + 显式状态管理 → 确定性流程
- AutoGPT：自主循环 → 不可控，不适合医疗
- LlamaIndex：侧重检索 → 缺少多轮对话编排

**关键细节：**
- 12 个节点 + 5 个条件边
- 支持 interrupt_before（人工介入）
- 状态自动持久化（Checkpoint）

**工程点：**
- 医疗场景需要确定性，不能"自由发挥"
- LangGraph 支持时间旅行调试
- 节点可独立测试

---

### 3. 多轮对话上下文管理

**结论：**
**短期窗口 + Checkpoint 持久化 + 历史裁剪**

**设计：**
- 短期：最近 N 轮对话（token 控制）
- 长期：LangGraph Checkpoint 持久化到 SQLite/PostgreSQL
- 裁剪：trim_history 节点自动截断超长历史

**关键细节：**
- 超过 token 限制自动裁剪
- 用户关键信息（症状/槽位）写入 state
- thread_id 隔离多会话

**工程点：**
- Redis 做 session 管理
- Checkpoint 支持断点恢复
- 支持多会话切换

---

### 4. 多 Agent 设计

**结论：**
采用**单图多节点编排**，暂不需要真正 Multi-Agent。

**设计：**
- 当前：LangGraph 单图多节点（decision/diagnosis/tool_calling/answer）
- 扩展：子图嵌套（Supervisor 模式）
- 通信：共享 State 对象

**关键细节：**
- 导诊场景单 Agent 足够
- 节点间通过 State 传递数据
- 条件边实现路由

**工程点：**
- LangGraph 支持子图嵌套
- Handoff 机制可转交任务
- 未来可扩展 Multi-Agent

---

### 5. 任务规划与状态回滚

**结论：**
**Checkpoint 快照 + 条件边回退**

**设计：**
- 每个节点执行后自动保存 Checkpoint
- 出错时回退到上一个稳定状态
- 条件边实现重试/降级路径

**关键细节：**
- LangGraph 内置时间旅行
- 支持 interrupt 后恢复
- 状态序列化存储

**工程点：**
- 节点独立 try/except
- 失败降级到 fallback
- 日志记录完整轨迹

## 二、RAG 与检索 (5题)

### 6. RAG 流程设计

**结论：**
**增强检索 + 重排 + 精简上下文**

**流程：**
1. query rewrite（纠错 + 扩展）
2. recall（ES + Milvus 并发）
3. rerank（bge-reranker-v2-m3）
4. context 构造（topK 裁剪）
5. LLM 生成

**关键细节：**
- ES 走关键词召回，Milvus 走语义召回
- RRF 融合两路结果
- rerank 控制在 top 5

**工程点：**
- 并发检索（asyncio.gather）
- 超时控制（避免慢查询拖垮）
- 10 秒超时降级

---

### 7. 为什么用 RAG？

**结论：**
解决三件事：**幻觉、私有知识、成本**

**说明：**
- LLM 无法覆盖医院流程类知识
- 直接问会产生 hallucination
- 长 context 成本高

**工程点：**
- RAG 可以做缓存
- 可控性更强（可审计）
- 知识更新无需重训模型

---

### 8. 检索不到结果的处理

**结论：**
**三级降级策略**

**策略：**
1. RAG 失败 → 直接 LLM 回答
2. LLM 低置信度 → 提示补充信息
3. 最终 fallback → 固定话术（"建议到导诊台咨询"）

**工程点：**
- 错误日志记录 badcase
- 可用于数据飞轮优化
- 置信度阈值 0.65

---

### 9. 检索优化（噪声/召回率）

**结论：**
**混合检索 + Rerank + RRF 融合**

**设计：**
- ES：关键词精确匹配
- Milvus：语义模糊匹配
- RRF：融合两路排序

**关键细节：**
- RRF 公式：1/(60 + rank)
- rerank 过滤噪声
- 阈值过滤低相关文档

**工程点：**
- 并发检索降低延迟
- 向量索引定期重建
- 缓存热点查询

---

### 10. 向量库更新

**结论：**
**增量更新 + 定期重建**

**策略：**
- 新数据 → 实时 embedding 入库
- 老数据 → 定期重算
- 删除 → 按 ID 删除 + 重建索引

**关键点：**
- embedding 漂移处理
- 版本控制
- ES/Milvus 同步

**工程点：**
- 消息队列同步
- 变更日志记录
- 一致性校验

---

## 三、Tool Calling 与可靠性 (4题)

### 11. Tool Calling 实现

**结论：**
基于 **function calling schema + MCP 协议**实现

**流程：**
1. 定义 tool schema（JSON）
2. LLM 输出参数
3. 后端执行函数
4. 返回 ToolMessage 给 LLM

**关键点：**
- 参数校验（Pydantic）
- 超时控制（10 秒）
- 独立工具模块

**工程点：**
- MCP Server 统一调度
- 工具可热插拔
- 错误重试机制

---

### 12. 函数调用鲁棒性处理

**结论：**
**参数校验 + 重试 + 降级**

**设计：**
- Pydantic 校验 LLM 输出
- 格式错误 → 重试 1 次
- 仍失败 → 降级到默认值

**关键细节：**
- schema 严格定义类型
- 超时自动中断
- 错误日志记录

**工程点：**
- 工具独立进程
- 熔断器模式
- 限流保护

---

### 13. 安全性（Prompt Injection）

**结论：**
**输入过滤 + 权限隔离 + 审计日志**

**设计：**
- 输入层：敏感词过滤
- 执行层：工具权限最小化
- 审计层：完整操作日志

**关键细节：**
- 不执行用户输入的代码
- 数据库只读权限
- 危急规则优先

**工程点：**
- 输入 sanitization
- RBAC 权限控制
- 日志脱敏存储

---

### 14. 长记忆管理

**结论：**
**滑动窗口 + 摘要压缩 + 向量检索**

**设计：**
- 短期：最近 N 轮（滑动窗口）
- 长期：关键信息向量化存储
- 压缩：定期 summarize 历史

**关键细节：**
- 超过 token 自动 summarize
- 用户关键信息（症状/病史）持久化
- embedding cache 提升性能

**工程点：**
- Redis 存储 session
- Checkpoint 持久化
- 摘要质量监控

## 四、后端工程与性能 (5题)

### 15. FastAPI 并发模型

**结论：**
**async + IO 多路复用**

**说明：**
- LLM/DB 都是 IO 密集
- async 提升吞吐
- uvicorn workers 多进程

**关键细节：**
- asyncio.gather 并发检索
- semaphore 限制最大并发
- 连接池复用

**工程点：**
- 16 核 CPU 支持 500+ 并发
- GPU 推理不占 CPU
- 异步中间件

---

### 16. 流式输出

**结论：**
**SSE (Server-Sent Events) 实现 streaming**

**流程：**
- LLM token 逐步返回
- 后端 chunk 推送给前端
- 前端逐字渲染

**关键细节：**
- astream_events 捕获 token
- 降低首字延迟（TTFT <300ms）
- 断线重连支持

**工程点：**
- SSE 比 WebSocket 简单
- 支持进度显示
- 用户体验提升

---

### 17. 响应延迟优化

**结论：**
**四层优化：量化 + 缓存 + 并发 + 流式**

**方法：**
1. INT4 量化（显存降 75%）
2. 语义缓存（热点查询直接返回）
3. 并发检索（ES+Milvus 同时查）
4. 流式输出（首字<300ms）

**工程点：**
- vLLM continuous batching
- KV cache 复用
- prompt 压缩

---

### 18. 限流与稳定性

**结论：**
**限流 + 超时 + 重试 + 熔断**

**设计：**
- 限流：token bucket（20 请求/秒）
- 超时：每个节点独立控制
- 重试：指数退避（最多 2 次）
- 熔断：失败率>50% 触发

**工程点：**
- Redis 计数器限流
- 超时自动降级
- 健康检查端点

---

### 19. 可观测性 (Observability)

**结论：**
**日志 + Tracing + 指标监控**

**设计：**
- 日志：结构化 JSON 日志
- Tracing：trace_id 串联请求
- 指标：Prometheus + Grafana

**关键细节：**
- 记录每个节点输入输出
- 延迟分布（P50/P95/P99）
- 错误率告警

**工程点：**
- LangSmith 跟踪 Agent 轨迹
- 失败自动归档 badcase
- 仪表盘实时监控

---

## 五、模型与成本优化 (4题)

### 20. Token 成本优化

**结论：**
**三层优化：context 裁剪 + 缓存 + 小模型优先**

**方法：**
1. 减少 context（rerank top 5）
2. query rewrite 提升命中率
3. 7B 优先，72B 兜底

**工程点：**
- prompt 压缩
- cache 命中直接返回
- 语义缓存（相似 query 复用）

---

### 21. RAG 缓存

**结论：**
**三层缓存结构**

**设计：**
- query cache：相同问题直接返回
- embedding cache：避免重复向量化
- answer cache：热点结果缓存

**工程点：**
- Redis + TTL（1 小时）
- 命中率监控
- 缓存失效策略

---

### 22. vLLM 优化原理

**结论：**
**continuous batching + PagedAttention**

**说明：**
- 多请求共享 GPU
- KV cache 分页管理，避免碎片
- continuous batching 动态合并请求

**结果：**
- 吞吐提升 2-3 倍
- 延迟下降 50%

**工程点：**
- prefix caching 复用前缀
- gpu-memory-utilization 0.8
- max-num-batched-tokens 控制

---

### 23. 模型选型策略

**结论：**
**场景驱动 + 成本效益分析**

**设计：**
- 7B：意图识别/提取/追问（延迟敏感）
- 72B：复杂推理/摘要生成（质量敏感）
- 本地部署：数据隐私 + 成本可控

**关键细节：**
- Qwen2.5 中文医疗能力强
- AWQ INT4 量化成熟
- Apache 2.0 可商用

**工程点：**
- vLLM 原生支持
- DashScope 商业支持
- 模型热切换
