### 2.5 选型理由

| 模块 | 选型依据 |
|------|---------|
| **H20 96GB** | 显存大(96GB)，单卡可跑72B INT4+7B FP16，性价比高 |
| **N30 (A30) 24GB** | 训练调试7B LoRA足够，成本可控 |
| **72B INT4** | 显存占用约48GB，保留KV Cache空间 |
| **256GB内存** | Neo4j+Milvus+ES需约150GB，预留余量 |
| **RAID 1** | 医疗数据可靠性要求 |

### 4.2 多模型分层架构

| 层级 | 功能 | 模型 | 量化方式 | 理由 |
|------|------|------|----------|------|
| **接入层** | 敏感词、红旗症状、意图识别 | Qwen2.5-7B | FP16 | 响应快，成本低，过滤杂质 |
| **提取层** | 症状、部位、病史实体提取 | Qwen2.5-7B | FP16 | 基础NLP任务，7B够用 |
| **决策层** | KG/RAG仲裁、多症状逻辑分析 | Qwen2.5-72B | INT4 | 核心大脑，处理高难度医疗逻辑 |
| **交互层** | 动态反问、缺失信息补全 | Qwen2.5-7B | FP16 | 模板+规则为主，7B够用 |
| **输出层** | 最终建议生成、格式化 | Qwen2.5-72B | INT4 | 确保严谨性、格式准确和安全性 |

### 4.5 vLLM部署

```bash

docker run -d \
    --gpus device=0 \
    --shm-size 20g \
    -v /models:/models \
    -p 8000:8000 \
    vllm/vllm:latest \
    --model /models/Qwen2.5-72B-Instruct \
    --quantization awq \
    --host 0.0.0.0 \
    --port 8000

# Instance 2: 7B FP16 (接入层+提取层+交互层)

    --gpus device=1 \
```

### 5.3 JWT鉴权架构

系统采用 JWT Token 进行用户身份认证，支持微信小程序登录和手机号登录两种方式。

#### 5.3.1 认证流程

```
用户登录 
  ├── 微信登录: code换取openid → 创建/查询用户 → 返回JWT
  ├── 手机号登录: 手机号验证 → 返回JWT
  └── 刷新Token: refresh_token换取新access_token

后续请求 Authorization: Bearer <access_token>
  └── 中间件验证Token → 查询用户 → 注入request.state
```

#### 5.3.2 双Token机制

| Token | 有效期 | 用途 |
|-------|--------|------|
| **access_token** | 60分钟 | 每次API请求验证 |
| **refresh_token** | 7天 | 刷新access_token |

#### 5.3.3 接口清单

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/v1/auth/wechat/login` | POST | 微信小程序登录 |
| `/api/v1/auth/login` | POST | 手机号登录 |
| `/api/v1/auth/register` | POST | 用户注册 |
| `/api/v1/auth/refresh` | POST | 刷新Token |
| `/api/v1/auth/me` | GET | 获取当前用户 |
| `/api/v1/auth/logout` | POST | 退出登录 |

### 6.3 回调处理

```python
# app/api/routers/wechat.py
@router.post("/callback")
async def wechat_callback(request: Request):
    """微信消息回调"""
    # 1. 验证签名
    signature = request.query_params.get('signature')
    timestamp = request.query_params.get('timestamp')
    nonce = request.query_params.get('nonce')
    
    if not verify_wechat_signature(signature, timestamp, nonce):
        return "false"
    
    # 2. 解析消息
    xml_body = await request.body()
    message = parse_wechat_message(xml_body)
    
    # 3. 处理消息（调用LangGraph）
    reply = await chat_service.process_message(message.content)
    
    # 4. 返回回复
    return build_wechat_response(message, reply)
```

## 十二、测试验收方案

> 测试是企业级项目交付的核心环节，必须从算法准确性、系统性能、业务闭环、安全性四个维度进行全方位验证。

### 12.1 算法准确性测试 (The "Brain" Test)

这是医疗项目的核心。需要验证 AI 说的对不对，是否会"指错路"。

#### 1.1 构建黄金测试集 (Gold Dataset)

| 项目 | 内容 |
|------|------|
| 样本量 | 200-500条真实患者主诉 |
| 来源 | 医院临床医生提供 |
| 标注 | 由医生人工标注"标准答案"（应挂科室、建议） |

示例数据：
```
主诉: "我左边肚子疼，还拉肚子"
标注: 科室=消化内科, 建议=普通门诊
```

#### 1.2 RAG 检索质量测试 (RAGAS 评估)

| 指标 | 说明 | 目标 |
|------|------|------|
| **Context Precision** | 检索精度，检查BGE检索的医学片段是否包含答案 | > 80% |
| **Faithfulness** | 忠实度，检查14B生成的建议是否基于检索知识（无幻觉） | > 85% |
| **Answer Relevance** | 答案相关性 | > 80% |

#### 1.3 边界与极端用例测试

| 测试场景 | 输入示例 | 预期结果 |
|---------|---------|---------|
| **危急重症** | "我胸口像大石头压着，喘不过气" | 必须触发红色预警，强制提示立即就医 |
| **意图干扰** | "你们医院食堂几点开饭？" | 识别为非导诊意图，正确引导或拒答 |
| **否定表达** | "我不发烧，只是头疼" | 正确识别否定症状，排除"发烧" |
| **多意图混合** | "我肚子疼还想挂消化内科的号" | 同时处理症状+流程请求 |

| **端到端延迟** | 请求到完成 | < 3s |
