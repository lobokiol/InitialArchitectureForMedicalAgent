# 企业级医院智能导诊系统需求规划

> 基于 COSTAR 框架编写
> 文档目标：将教学型 RAG 项目落地为医院正式上线的智能导诊/分诊系统

---

# COSTAR 框架

## C - Context（背景）

### 1.1 项目背景

**项目名称**：Hospital_guidance_agent  
**当前形态**：教学型/原型级 RAG 项目  
**核心技术栈**：FastAPI + LangGraph + Redis + Elasticsearch + Milvus + DashScope  
**核心能力**：基于语义匹配的医疗问答、流程指引、多轮对话

### 1.2 使用场景

**场景描述**：三级甲等医院门诊/急诊入口处，患者通过自助终端或手机小程序进行预检分诊

**目标用户**：
- 门急诊患者（尤其自助就医群体）
- 陪诊人员
- 导诊护士

**核心价值**：
- 减轻导诊人工压力 60%+
- 实现 7×24 小时智能分诊
- 覆盖 85%+ 常见症状

### 1.3 教学型 vs 企业级对比

| 维度 | 教学型架构（当前） | 企业级导诊系统 |
|------|-------------------|----------------|
| **核心逻辑** | 语义搜索 + LLM 生成 | 医学临床指南 + 风险等级判定 |
| **数据源** | 本地静态 PDF/Markdown | 实时 HIS 数据库 + 动态排班表 |
| **交互形式** | 纯文本聊天 | 语音 + 3D 人体图 + 多端适配 |
| **准确性** | 依靠 LLM 概率生成 | 医学图谱约束 + 医生审核库 |
| **安全责任** | 仅供学习参考 | 医疗事故追溯 + 严控敏感信息 |
| **系统集成** | 孤立聊天框 | 与 HIS/EMR/LIS 深度集成 |
| **分诊标准** | 简单语义匹配 | ESI 标准 / 国内五级分类法 |

## O - Objective（目标）

### 2.1 核心业务目标

| 优先级 | 功能目标 | 成功指标 |
|--------|----------|----------|
| P0 | 智能症状问诊与科室推荐 | 分诊准确率 ≥ 92%（首批上线目标 85%） |
| P0 | 精准号源查询与挂号引导 | 实时对接 HIS 排班，号源准确率 100% |
| P1 | 急诊分级预检 | 符合 ESI 四级分类标准 |
| P1 | 流程指引与就医路线 | 科室位置导航准确率 ≥ 95% |
| P2 | 复诊与随访提醒 | 就医闭环管理 |
| P2 | 投诉与反馈闭环 | 满意度提升 20% |

### 2.2 非功能性目标

| 指标 | 要求 |
|------|------|
| **响应时间** | 首字响应 < 2s（TP99 < 5s） |
| **并发能力** | 支持 500+ 同时在线用户 |
| **可用性** | 99.9% SLA，7×24 小时服务 |
| **日活** | 支持 10000+ 问诊请求/天 |

## S - Scenario（场景）

### 3.1 主场景：智能分诊

```
患者入口 → 症状描述（文本/语音/人体图） → AI 问诊追问 
→ 风险评估（急诊/普通） → 科室推荐 → 号源查询 
→ 挂号确认 → 就医指引
```

### 3.2 用户故事

| 角色 | 故事 | 期望 |
|------|------|------|
| **患者A** | 我喉咙痛，不知道挂什么科 | 系统推荐耳鼻喉科，显示当前号源 |
| **患者B** | 我胸口剧痛，呼吸困难 | 系统识别为危急症状，强制提示挂急诊 |
| **患者C** | 老人家行动不便 | 系统推荐楼层有电梯的科室，提供轮椅租赁指引 |
| **患者D** | 复查配药 | 系统识别复诊患者，推荐原就诊科室和医生 |

### 3.3 关键子流程

| 子流程 | 描述 | 异常处理 |
|--------|------|----------|
| **症状采集** | 支持主诉选择、补充描述、体征输入 | 模糊描述触发追问，未明确症状引导转人工 |
| **风险评估** | 结合生命体征（血压/心率/体温）判定危急程度 | 高风险直接提示"请立即就医"并联动急诊 |
| **科室匹配** | 症状→疾病→科室 知识图谱推理 | 复杂病症推荐多科室或转人工 |
| **号源查询** | 实时查询科室/医生排班、剩余号源 | 无号源推荐相邻时段或候补 |
| **挂号确认** | 跳转至医院挂号平台 | 挂号失败保留就诊建议记录 |

## T - Task（任务）

### 4.1 核心任务清单

#### 4.1.1 智能问诊引擎

| 任务 | 描述 | 技术要求 |
|------|------|----------|
| **多轮问诊** | 支持 5-10 轮追问，问清关键症状要素 | 状态机管理，问诊进度可视化 |
| **意图识别** | 区分主诉、问路、医保咨询、闲聊 | 多分类模型 + 规则兜底 |
| **症状理解** | 理解"肚子疼"=腹痛，"胸闷"=胸闷感 | 医学 NLP 处理，同义词映射 |
| **体征输入** | 支持血压、心率、体温等数值输入 | 正则校验 + 范围判定 |
| **风险评估** | 基于 ESI 标准分级（1-4 级） | 规则引擎 + LLM 综合判定 |

#### 4.1.2 知识库与 RAG

| 任务 | 描述 | 技术要求 |
|------|------|----------|
| **医学知识图谱** | 症状-疾病-科室-指南 推理链路 | Neo4j / TuGraph，构建知识网络 |
| **Medical Embedding** | 医疗领域专用向量模型 | BioBERT / 医学微调模型 |
| **多源知识融合** | 临床指南 + 药品说明 + 医院制度 | 结构化 + 非结构化混合检索 |
| **知识溯源** | 答案标注来源，置信度显示 | 检索结果打分，前 N 条引用 |

#### 4.1.3 HIS/EMR 集成

| 任务 | 描述 | 协议方式 |
|------|------|----------|
| **患者画像** | 获取既往病史、过敏史、就诊记录 | FHIR / HL7 / 数据库视图 |
| **号源查询** | 实时科室/医生排班、剩余号源 | HIS API 对接 |
| **挂号跳转** | 成交挂号记录回传 | 回调接口 |
| **报告查询** | 检查/检验报告结果推送 | LIS 接口 |

#### 4.1.4 安全与合规

| 任务 | 描述 | 技术要求 |
|------|------|----------|
| **内容审核** | 敏感词过滤，有害内容拦截 | 医学词典 + 规则引擎 + LLM Guardrails |
| **风险提示** | 高危症状强制提示"请立即就医" | 硬编码规则 + LLM 判断双保险 |
| **数据脱敏** | 患者信息 PII 脱敏处理 | 等保三级要求，加密存储 |
| **操作审计** | 完整日志留痕，可追溯 | 审计日志 + 操作记录 |
| **免责声明** | 明确告知"AI 建议仅供参考" | 前端强提示 + 确认机制 |

## A - Action（行动）

### 5.1 系统架构设计

```
┌─────────────────────────────────────────────────────────────────┐
│                        接入层 (API Gateway)                      │
│   小程序 / 自助终端 / 微信公众号 / 医院 APP / Web 门户          │
└─────────────────────────────────────────────────────────────────┘
                                  ↓
┌─────────────────────────────────────────────────────────────────┐
│                      业务中台 (Business Layer)                   │
│  ┌──────────────┐ ┌──────────────┐ ┌──────────────────────┐   │
│  │  智能分诊引擎  │ │  号源查询服务  │ │    知识库管理服务     │   │
│  │  (LangGraph) │ │  (HIS 集成)   │ │ (RAG + 知识图谱)      │   │
│  └──────────────┘ └──────────────┘ └──────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
                                  ↓
┌─────────────────────────────────────────────────────────────────┐
│                      数据层 (Data Layer)                         │
│  ┌────────┐ ┌─────────┐ ┌────────┐ ┌────────┐ ┌────────────┐  │
│  │ HIS    │ │ EMR     │ │ LIS    │ │ 知识库  │ │  向量库    │  │
│  │ (排班) │ │ (病历)  │ │ (检验) │ │ (指南)  │ │ (Milvus)   │  │
│  └────────┘ └─────────┘ └────────┘ └────────┘ └────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

### 5.2 技术选型

| 模块 | 技术方案 | 说明 |
|------|---------|------|
| **LLM** | qwen-max + 本地模型 | 高并发场景保障，降低 API 成本 |
| **向量库** | Milvus Cluster / pgvector | 生产级高可用，支持亿级向量 |
| **搜索** | Elasticsearch + 混合检索 | 全文+向量混合排序 |
| **状态管理** | Redis Cluster | 高可用 + 大规模会话 |
| **Agent 框架** | LangGraph + 自研工作流 | 复杂分诊流程编排 |
| **知识图谱** | Neo4j / TuGraph | 症状-疾病-科室推理 |

### 5.3 核心 API 接口

| 接口 | 方法 | 描述 |
|------|------|------|
| `/api/v1/triage/start` | POST | 开启分诊会话 |
| `/api/v1/triage/chat` | POST | 症状问答（流式返回） |
| `/api/v1/triage/result` | GET | 获取分诊结论 |
| `/api/v1/department/query` | GET | 科室列表查询 |
| `/api/v1/doctor/schedule` | GET | 医生排班与号源 |
| `/api/v1/appointment/register` | POST | 挂号预约 |
| `/api/v1/feedback` | POST | 满意度反馈 |
| `/api/v1/admin/audit` | GET | 分诊记录审计（管理员） |

### 5.4 数据规模规划

| 类型 | 条目数量（估计） | 说明 |
| ----------- | -------------- | ----------------------- |
| **症状问答** | ~200,000–800,000 | 常见症状与就诊指引（如发烧、头痛、消化不良等） |
| **科室与就诊流程** | ~1,000–3,000 | 各科室职责、预约流程、就诊流程 |
| **疾病与诊疗指南** | ~5,000–20,000 | 常见病种诊断、流程、科室指引 |
| **检查/检验说明** | ~5,000–15,000 | CT/MRI/化验/超声等检查注意事项及流程 |
| **药物信息** | ~3,000–8,000 | 常用处方/OTC 药物说明、剂量、禁忌 |
| **科室信息元数据** | ~100–300 | 所有临床及医技科室信息 |
| **政策与辅助内容** | ~2,000–10,000 | 挂号、医保、费用、住院指南、健康宣教 |

## R - Result（结果）

### 6.1 产出物

| 产出物 | 描述 |
|--------|------|
| **分诊结论** | 推荐科室、急诊/普通标识、优先级 |
| **就医建议** | 检查项目、注意事项、准备材料 |
| **挂号入口** | 跳转医院官方挂号小程序/APP |
| **导航信息** | 科室位置、楼层、预计步行时间 |
| **知识库引用** | 依据的诊疗指南来源（可选展示） |

### 6.2 角色与责任

| 角色 | 职责 |
|------|------|
| **患者/用户** | 输入症状、确认信息、完成挂号、评价服务 |
| **AI 导诊系统** | 症状问诊、分诊建议、流程指引、号源查询 |
| **导诊护士（人工兜底）** | 复杂病例接管、AI 异常case处理、投诉处理 |
| **科室医生（审核）** | 分诊结果抽检、纠错反馈、规则优化 |
| **运维/管理员** | 系统监控、知识库更新、报表查看 |

### 6.3 成功标准

| 指标 | 目标值 |
|------|--------|
| 分诊准确率 | ≥ 92%（首批 ≥ 85%） |
| 号源准确率 | 100% |
| 导航准确率 | ≥ 95% |
| 满意度提升 | 20% |
| 人工替代率 | 60%+ |

## 实施路线

### 阶段一：MVP（第 1-2 个月）

- [ ] 智能症状问诊 + 科室推荐（基于现有 RAG 增强）
- [ ] 简单流程指引（科室位置、注意事项）
- [ ] 基础版 Web 前端
- [ ] 接入公开知识库（临床指南、药品说明）

### 阶段二：成长（第 3-4 个月）

- [ ] HIS 号源查询集成
- [ ] 医学知识图谱构建
- [ ] 急诊 ESI 分级预检
- [ ] 人工兜底通道（转人工客服）

### 阶段三：成熟（第 5-6 个月）

- [ ] EMR 患者画像接入
- [ ] 复诊随访管理
- [ ] 医生反馈闭环（RLHF）
- [ ] 管理后台与数据分析
- [ ] 多端覆盖（小程序、终端、公众号）
